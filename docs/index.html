The problem

What existing tools miss entirely

OWASP DockSec wraps Trivy, Hadolint, and Docker Scout with AI explanations — and does it well. But the entire class of runtime and composition misconfigurations is invisible to it. DockerDNA was built to fill those gaps.

No compose scanner

DockSec only accepts a Dockerfile. Privileged mode, Docker socket mounts, missing resource limits, and host networking in docker-compose.yml are never flagged — not by DockSec, Trivy, or Hadolint.

No secrets detection

Trivy finds known CVEs; Hadolint checks syntax. Neither scans for hardcoded AWS keys, GitHub tokens, database URIs, or JWT tokens embedded across Dockerfiles and .env files.

No entropy analysis

Regex patterns miss custom or rotated credentials. Shannon entropy analysis is the only way to catch high-entropy strings that look like secrets but match no known pattern.

No CIS compliance score

DockSec produces severity buckets. DockerDNA maps every finding to a specific CIS Docker Benchmark v1.6 control and produces a pass/fail scorecard with a 0–100% compliance score.

No SARIF output

GitHub Advanced Security requires SARIF to annotate PRs in the Security tab. This has been an open DockSec issue (#45) since launch. DockerDNA ships SARIF 2.1.0 by default.

No SBOM generation

Supply chain transparency requires a Software Bill of Materials. DockerDNA parses every apt-get, pip, npm, apk, and yum install to produce a CycloneDX 1.5 BOM with PURL identifiers.

The four-scanner pipeline
1

Secrets Engine — Regex + Shannon Entropy

Scans Dockerfiles, docker-compose.yml, .env files, and any project files. Pattern matching covers 20+ known secret formats (AWS, GCP, GitHub, Stripe, JWT, DB URIs…). Entropy analysis with threshold ≥ 4.5 bits/char catches unknown high-entropy tokens that regex never will.

H(X) = -∑ p(x) log₂ p(x)  →  flag if H ≥ 4.5 and length ≥ 20 chars.

2

docker-compose.yml Scanner — CIS Section 5

Parses every service definition in YAML. Checks for privileged: true, cap_add: [ALL], Docker socket volume mounts, network_mode: host, missing memory/CPU limits, missing read_only, missing no-new-privileges, missing AppArmor profile, and hardcoded env secrets. Every finding is tagged with its CIS Docker Benchmark v1.6 control ID.

3

Dockerfile + Supply Chain Scanner — Layer Attribution

Parses the Dockerfile layer-by-layer. Detects :latest tags without digest pinning, unnecessary packages (curl, vim, gcc…), missing USER and HEALTHCHECK, secrets leaking across multi-stage builds, and SSH server installation. Each FROM image receives a 0–100 supply chain risk score based on registry trust, tag specificity, and Docker Content Trust status.

4

CIS Compliance Mapper + Report Generation

All findings are aggregated and mapped to CIS Docker Benchmark v1.6 controls. A pass/fail scorecard is produced. Reports are written in all requested formats: JSON (full structured data), HTML (interactive dashboard), SARIF 2.1.0 (GitHub Security tab), and CycloneDX 1.5 SBOM. A CI threshold gate (--threshold HIGH) exits with code 1 if findings at that severity or above are present.

secrets.py — Dual-Mode Detection

Two complementary strategies: compiled regex for 20+ known secret formats, and Shannon entropy analysis for unknown high-entropy tokens. Values are automatically redacted in reports.

aws-keys github-tokens jwt db-uris entropy≥4.5
📦

compose.py — Runtime Misconfig Scanner

YAML parser covering every CIS Section 5 control: privileged mode, capability escalation, Docker socket mounts, host networking, missing resource limits, read-only filesystem, and no-new-privileges.

CIS-5.3 CIS-5.4 CIS-5.9 CIS-5.13 CIS-5.14
🔗

supply_chain.py — Risk Scoring

Every FROM image receives a 0–100 supply chain risk score based on registry trust (official vs community vs self-hosted), digest pinning, Docker Content Trust status, and malicious image name patterns.

CIS-4.5 digest-pinning registry-trust content-trust
📋

compliance.py — CIS Scorecard

Aggregates all scanner findings and maps them to CIS Docker Benchmark v1.6 controls. Produces a full pass/fail/not-checked scorecard and a 0–100% compliance percentage.

sbom.py — CycloneDX SBOM

Parses every package manager install instruction (apt-get, pip, npm, apk, yum) to produce a CycloneDX 1.5 BOM with PURL identifiers, layer attribution, and build-stage annotations. NTIA minimum-elements compliant.

Detection coverage

What DockerDNA finds

Findings across five scanners, mapped to CIS Docker Benchmark v1.6 controls.

🔐

Hardcoded Secrets

AWS access keys, Google API keys, GitHub tokens, Stripe keys, JWT tokens, RSA private keys, database connection strings, and 15+ more patterns. Plus Shannon entropy detection for unknowns.

⚠

Privileged Containers

privileged: true and cap_add: [ALL] in docker-compose give a container unrestricted kernel access. CIS-5.4 and CIS-5.3. Flagged with specific remediation steps.

🔄

Docker Socket Mounts

Mounting /var/run/docker.sock gives a container full control over the Docker daemon — equivalent to root on the host. CIS-5.13. Flagged as CRITICAL.

👤

Root Container Execution

Containers without a non-root USER instruction run as UID 0. If the application is compromised, the attacker has root inside the container. CIS-4.1.

🏭

Supply Chain Risks

Base images without digest pinning, using Docker Hub community images, or without Docker Content Trust enabled receive elevated risk scores. CIS-4.2 and CIS-4.5.

📋

Multi-Stage Secret Leaks

Secrets set in ENV or ARG in early build stages may survive into the final image layer history. DockerDNA detects this pattern across all stages and warns before the image is built.

💸

Missing Resource Limits

Containers without memory or CPU limits can consume all host resources, causing denial of service for other workloads. CIS-5.10 and CIS-5.11.

🔌

Missing Security Controls

No HEALTHCHECK (CIS-4.6), no read-only root filesystem (CIS-5.12), no no-new-privileges (CIS-5.14), no AppArmor profile (CIS-5.1). All flagged with CIS control IDs and remediations.

🔬

Unnecessary Packages

curl, wget, vim, gcc, gdb, strace, openssh-server, and 10+ other tools that expand the attack surface when present in production images. CIS-4.3.

Installation

Get started in 30 seconds

Install via pip, point at your Dockerfile or project directory, get results in JSON, HTML, SARIF, and SBOM.

● 1. Install
pip install dockerdna

# With AI remediation
pip install "dockerdna[ai]"
Requires Python 3.11+. PyYAML is the only non-stdlib dependency for core scanning.
● 2. Scan your project
dockerdna Dockerfile \
  --compose docker-compose.yml \
  --format json html sarif sbom
Output written to ./dockerdna-results/. HTML report includes an interactive CIS compliance scorecard.
● 3. CI/CD gate
dockerdna Dockerfile \
  --compose docker-compose.yml \
  --threshold HIGH \
  --format sarif
Exits with code 1 if any CRITICAL or HIGH findings are present. Use in GitHub Actions to block unsafe PRs.
● 4. AI remediation
export ANTHROPIC_API_KEY=sk-...
dockerdna Dockerfile \
  --compose docker-compose.yml \
  --ai --format json html
Claude synthesises findings from all five scanners and produces a prioritised fix plan with corrected Dockerfile and compose file.
GitHub Actions integration
● .github/workflows/security.yml
- uses: actions/checkout@v4
- run: pip install dockerdna

- name: Run DockerDNA
  run: |
    dockerdna Dockerfile \
      --compose docker-compose.yml \
      --format sarif json \
      --threshold HIGH \
      --output dockerdna-results

- name: Upload to GitHub Security tab
  uses: github/codeql-action/upload-sarif@v3
  if: always()
  with:
    sarif_file: dockerdna-results/report.sarif
SARIF findings appear as inline annotations on pull requests in the GitHub Security tab. No additional configuration required.
